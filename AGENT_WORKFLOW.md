# AGENT_WORKFLOW.md

How this project was built with AI tools — what the tools did, where I steered,
where they failed, and how a teammate can audit the work.

> This is a reviewer-facing companion to the code. My private, running log of the
> process is in `docs/Implementaion_process_user exclusive.txt`; the single best
> steering example is in [`docs/best_interaction_artifact.md`](docs/best_interaction_artifact.md).

---

## 1. Tools used

| Tool | Role |
|---|---|
| **ChatGPT** | Up-front brainstorming: scoping the problem, comparing architectures, pressure-testing my understanding, and generating the first drafts of the source-of-truth docs (`product_requirements.md`, `system_architecture.md`, `system_flow.md`). |
| **A second LLM** | Independent cross-review of those docs before I froze them. |
| **Claude Code (CLI)** | The implementation agent: wrote the code, tests, and most docs — always driven by the approved documents and gated by my review, phase by phase. Used with plan-mode (plan → I approve → implement → I review → commit). |
| **MCP-style tooling / deterministic dev sandbox** | File, search, shell, and git tooling the agent used to edit, run tests, run the app, and verify behaviour. |

I did **not** one-shot this. The workflow was deliberately **architecture-first**:
freeze the specs, then let the agent implement against them in small, reviewable
steps.

## 2. Operating model (how I steered)

1. **Docs are the source of truth.** The three `docs/*.md` specs and `AGENTS.md`
   (immutable project rules) drove every step. When implementation hit ambiguity,
   the rule was *ask, don't invent architecture*.
2. **Plan-mode gate.** Each phase started in plan mode: the agent explored the
   code and produced a written plan; I reviewed/changed it; only then did it code.
3. **Phase-by-phase, one commit per phase.** Six phases (0–5), each reviewed and
   committed separately — see `git log`. Post-MVP work (Docker, runtime LLM
   provider selection) followed the same plan→review→commit loop.
4. **Human approval before anything lands.** Nothing was committed without my
   review. I rejected and reverted output that wasn't good enough (see §4).

## 3. AI-generated vs. manually steered

**Almost all code and test text was generated by the agent** — but under tight
human direction. The judgment was mine:

| Mine (human) | Agent-generated (reviewed by me) |
|---|---|
| Problem framing, users, scope, and the decision to mock Layer 1 | The Python implementation of every workflow node, service, schema, and test |
| The approved architecture and data contracts (docs) | LangGraph graph wiring, FastAPI routers, Gradio UI, renderer templates |
| Every phase plan review; accept / reject / revise | First drafts of the phase plans |
| Product decisions: chat-first recommendation, live progress tracker, single-container packaging, **Mock as the default provider**, **in-memory-only keys**, `docker/` layout | The mechanics that implement each decision |
| Catching wrong output and directing the fix | The fixes themselves |
| Wording/ownership of the narrative docs (this file, README framing) | First drafts of those docs |

Concrete design calls I made that shaped the code:
- **Deterministic fallback is mandatory**, so the app runs with no key/network.
- **Evidence-first**: no recommendation without resolvable evidence citations.
- **Human-in-the-loop**: no external action without explicit approval.
- **UI is presentation-only**; all reasoning/business logic stays in the backend.
- The runtime LLM-provider switch had to **reuse the existing provider seam**, not
  redesign anything.

## 4. Where the agent helped — and where it failed

**Helped most:** scaffolding a consistent multi-module codebase fast; keeping the
LangGraph/FastAPI/Gradio layers wired correctly; generating thorough tests
alongside features; and doing careful, mechanical refactors (e.g. adding a second
LLM provider through the existing abstraction).

**Failed / had to be corrected (all caught in review):**
- **Committed `.env` with a real API key.** Caught it; removed from tracking,
  rotated/scrubbed. The repo now commits only `.env.example`; `.env` is gitignored.
- **A "continue" attempt produced broken wiring** (missing template, inconsistent
  imports). I **reverted it** and had the work redone consistently.
- **A modification instruction was ignored** — asking to change the draft email
  didn't change it. Root cause: the human feedback never reached the draft-
  generation node. Fixed by injecting the revision only at the resume-target node.
- **Renderer template whitespace bug** (Jinja `trim_blocks` ran action items
  together) — fixed by using inline expressions.
- **Windows console `UnicodeEncodeError`** on an em-dash — fixed by reconfiguring
  stdout to UTF-8.
- **A flaky test** from Starlette `TestClient` not being concurrency-safe with the
  UI's polling — fixed by restructuring the poll loop.
- **Shipped-image staleness** — the first Docker image predated the "Mock default"
  change; I caught it before sending and rebuilt from the committed code.

The pattern: the agent moved fast and I owned correctness — reviewing every diff,
running the app, and rejecting anything that didn't hold up.

## 5. How a teammate should audit this work

1. **Read the specs first:** `docs/product_requirements.md`,
   `system_architecture.md`, `system_flow.md`, and `.claude/AGENTS.md`. The code is
   meant to match these.
2. **Walk the git history:** `git log --oneline`. Phases 0–5 are separate,
   reviewable commits; post-MVP work follows.
3. **Run it offline (no key):**
   ```bash
   docker compose up            # or: docker build -f docker/Dockerfile -t hiring-agent . && docker run -p 7860:7860 -p 8000:8000 hiring-agent
   ```
   Open http://localhost:7860, start a session, and exercise approve / modify /
   off-topic. It runs on the deterministic **Mock** reasoner by default.
4. **Run the checks:**
   ```bash
   python -m pytest backend/tests -q     # 100 tests, offline
   python -m ruff check backend frontend
   ```
5. **Spot-check the invariants the docs promise:**
   - *Evidence-first* — every recommendation cites resolvable evidence (see the
     Internal Processing panel and `services/evidence.py`).
   - *Human-in-the-loop* — the graph interrupts before `wait_for_human`; no mock
     execution runs before approval (`workflow.py`, `nodes/approve.py`).
   - *UI has no business logic* — `frontend/` only makes HTTP calls
     (`frontend/api_client.py`); reasoning lives in `backend/`.
   - *Secrets* — grep the repo for keys; only `.env.example` ships, and UI-entered
     keys are in-memory only (never logged or returned — see
     `backend/core/llm/runtime.py` and the masked `GET /api/v1/llm/config`).
6. **Read the steering artifact:** [`docs/best_interaction_artifact.md`](docs/best_interaction_artifact.md)
   shows one end-to-end human-steered change (runtime LLM-provider selection).
