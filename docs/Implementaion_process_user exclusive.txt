This doc is for monitoring how I am implementing this project. Don't use this file as a reference or edit it.

Before starting development, I had multiple brainstorming sessions with ChatGPT to define the project scope, validate my understanding, discuss different architectural approaches, identify potential issues, and resolve my doubts. Once the overall direction was finalized, I asked it to generate detailed project documents that would act as the source of truth for implementation instead of relying on vague prompts during development.

The generated documents were reviewed by me and cross-validated with another LLM. After several iterations and improvements, the final versions were approved. These include `product_requirements.md`, `system_architecture.md`, and `system_flow.md`, which together define the expected behavior, architecture, workflows, and implementation boundaries of the MVP.

Link to the design discussion:
https://chatgpt.com/share/6a46a03e-e13c-83e8-8ed2-09fba4329648

These finalized documents have been saved in the `docs/` folder at the root of the project. They serve as the primary reference for all future development decisions.

The implementation follows an architecture-first approach. Instead of trying to build the application using one-shot prompts, Claude Code is used as the implementation agent. Every implementation step is driven by the approved documentation, allowing each phase to be reviewed before moving to the next. This keeps development structured, reduces ambiguity, and minimizes scope drift.

The current phase is setting up the project foundation, including the folder structure, `.agent.md` files, development workflow, technology stack, and project scaffold. Once this foundation is finalized, development will proceed feature by feature according to the defined system.

### Current Progress

**Step 1 — Completed:** Using Claude Code as the implementation agent, I completed repository initialization and planning. Claude Code read the approved source documents (`product_requirements.md`, `system_architecture.md`, `system_flow.md`) and `AGENTS.md`, analyzed the mock meeting packages, hiring tracker, and candidate profiles, then produced the engineering plan under `Implementation_docs/` — `MASTER_IMPLEMENTATION.md` (system overview, component breakdown, dependency graph, risks, validation strategy, and per-phase specs) and `TRACKER.md` (living tracker, all phases NOT STARTED). It also bootstrapped the repo (`.venv` created without installing packages, `requirements.txt`, `.env.example`, `.gitignore`) and appended an Implementation Workflow section to `AGENTS.md`. The work is broken into five reviewable implementation phases. No application code was written in this step.

**Step 2:** Begin phase-by-phase implementation — draft a plan for each phase in plan mode, review it, then execute only after approval.

**Step 3 — Completed:** With all five implementation phases finished and the
MVP working end to end, I shifted focus to packaging and portability so the
project can be evaluated with minimal setup. Using Claude Code as the
implementation agent, I had the application containerized as a single Docker
image that runs both the FastAPI backend and the Gradio UI. The image defaults
to the deterministic offline ("Mock") reasoner, so it starts and completes a
full session with no API key and no network. I reviewed the setup and had the
container build files consolidated under a `docker/` folder (`docker/Dockerfile`,
`docker/entrypoint.sh`), keeping `.dockerignore` and `.gitattributes` at the repo
root where they belong.

**Step 4 — Completed:** To let a reviewer optionally try live reasoning
without being tied to a single vendor, I asked for the LLM layer to be made
provider-switchable at runtime from the UI. The result is a single "Provider"
control — Mock (default), Ollama Cloud, or Azure OpenAI — surfaced in an
"LLM Settings" panel. Selecting a provider and applying it calls a backend
endpoint that rebuilds the LLM client; any API key is entered in the UI and held
in memory only (never written to disk or logged, and only ever reported back
masked). This respected the existing architecture — the provider abstraction was
already present, so no redesign was needed — and all reasoning/business logic
stayed in the backend, with the UI only forwarding the selection. The change is
covered by tests (a new Azure provider unit test and an LLM config API test), and
the full suite stays green with the linter clean.

**Delivery:** For evaluation I chose to share the repository so the reviewer
needs only Docker: they clone the repo and run
`docker build -f docker/Dockerfile -t hiring-agent .` followed by
`docker run -p 7860:7860 -p 8000:8000 hiring-agent`, then open the UI at
`http://localhost:7860`. It runs offline in Mock mode by default; switching to
Ollama or Azure is optional and done entirely from the UI. The README and
`.env.example` document the providers, the Azure configuration keys, and the
in-memory key handling.
